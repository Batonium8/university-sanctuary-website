# client.js, router.js, useBooking.js — «Нервная система» приложения

Если компоненты (Header, BookingForm) — это «мышцы» и «кожа» сайта, то эти три файла — его **нервная система и кровеносные сосуды**. Они отвечают за то, как приложение общается с внешним миром, как пользователь перемещается между страницами и как компоненты обмениваются данными без прямой связи.

---

## 1. client.js — Сетевой слой (Axios)

Этот файл создаёт **единый настроенный экземпляр** для HTTP-запросов. Вместо того чтобы в каждом сервисном файле (`homes.js`, `services.js`) настраивать запросы с нуля, мы делаем это один раз здесь.

### Ключевые настройки

```javascript
const apiClient = axios.create({
  baseURL: '/sanatorium/api/v1',
  timeout: 10000,
  headers: { 'Content-Type': 'application/json', 'Accept': 'application/json' },
});
```

| Настройка | Зачем нужна |
|-----------|-------------|
| `baseURL` | **DRY (Don't Repeat Yourself)**. Мы пишем `apiClient.get('/homes')`, а Axios сам склеивает это с базовым URL. Если бэкенд переедет на другой домен, мы поменяем только одну строчку. |
| `timeout` | Защита от зависаний. Если сервер молчит больше 10 секунд, запрос обрывается, и пользователь не ждёт вечно. |
| `headers` | Говорим серверу: «Мы шлём JSON и хотим получить JSON в ответ». |

### Магия интерцепторов (Interceptors)

```javascript
apiClient.interceptors.response.use(
  (response) => response,
  (error) => {
    console.error('API Error:', error.response?.status);
    return Promise.reject(error);
  }
);
```

**Интерцептор** — это «перехватчик». Он стоит между вашим кодом и сервером.
* **Теория:** Раньше, чтобы обработать ошибку сети или 401 (Unauthorized), пришлось бы писать `try/catch` в каждом компоненте. С интерцептором мы ловим **все** ошибки глобально в одном месте. В будущем сюда можно добавить автоматический редирект на страницу логина, если токен протух.

## 2. router.js — Маршрутизация (SPA Навигация)

Этот файл превращает набор Vue-компонентов в настоящее **SPA (Single Page Application)** с реальными URL-адресами.

### Конфигурация истории

```javascript
history: createWebHistory(import.meta.env.BASE_URL)
```
* **`createWebHistory`** — использует HTML5 History API. URL выглядят как обычные: `/rooms/1`, а не как `/rooms/1#/hash`. Это критично для SEO, так как поисковики лучше индексируют чистые URL.
* **`import.meta.env.BASE_URL`** — переменная окружения. Если сайт будет лежать в подпапке (например, `site.ru/app/`), роутер сам добавит этот префикс ко всем ссылкам.

### Динамические сегменты и 404

```javascript
{ path: '/rooms/:id', name: 'HomeDetail', component: HomeDetailsView },
{ path: '/:pathMatch(.*)*', name: 'NotFound', component: NotFound }
```
* **`:id`** — динамический параметр. Роутер понимает, что вместо `:id` может быть любое число или строка, и передаёт это значение в компонент через `route.params.id`.
* **Catch-all route (`/:pathMatch(.*)*`)** — «ловец» всех несуществующих маршрутов. Если пользователь введёт `/абракадабра`, роутер не упадёт с белым экраном, а покажет красивую страницу 404. *Важно: этот маршрут должен быть строго последним в массиве.*

### Глобальное поведение скролла

```javascript
scrollBehavior(to, from, savedPosition) {
  return { top: 0, behavior: 'smooth' };
}
```
По умолчанию Vue Router запоминает позицию скролла. Но для нашего сайта мы решили: **при любом переходе между страницами плавно (`smooth`) скроллить вверх (`top: 0`)**. Это улучшает UX, чтобы пользователь не начинал читать новую страницу с середины.

## 3. useBooking.js — Глобальное состояние (Composable)

Этот файл решает классическую проблему архитектуры Vue — **Prop Drilling** (пробрасывание пропсов через 10 слоёв компонентов).

### Проблема, которую он решает
Представь: компонент `HomeDetail` (глубоко в недрах страницы) загрузил данные о номере. А компонент `BookingForm` (в другом конце DOM-дерева, в App.vue) должен знать ID этого номера, чтобы отправить его на сервер.
Передавать `id` через пропсы: `HomeDetail -> HomeDetailsView -> App -> BookingForm` — это грязно и неудобно.

### Решение: Composable (Shared State)

```javascript
export const selectedHomeId = ref(null);

export function setSelectedHomeId(id) {
  selectedHomeId.value = id ? Number(id) : null;
}
```

**Как это работает:**
1. Мы создаём обычную Vue-переменную `ref()` и **экспортируем её**.
2. Во Vue 3, если экспортировать `ref` из модуля, он становится **глобальным реактивным состоянием**. Все компоненты, которые импортируют `selectedHomeId`, ссылаются на **один и тот же объект в памяти**.
3. Когда `HomeDetail` вызывает `setSelectedHomeId(1)`, переменная меняется. `BookingForm`, который просто "слушает" эту переменную (`home_id: selectedHomeId.value`), мгновенно узнаёт об изменении.

**Почему это лучше Pinia/Vuex?**
Для хранения одного ID номера нет смысла тянуть в проект тяжёлую библиотеку управления состоянием (вроде Pinia). Простой composable с `ref` работает точно так же, но не требует настройки, сторов и бойлерплейт-кода.

## 🧠 Итоговая архитектура (Как они связаны)

```text
[Пользователь кликает "Номера"]
      │
      ▼
[router.js] → меняет URL на /rooms, монтирует компонент
      │
      ▼
[Компонент Rooms.vue] делает запрос
      │
      ▼
[homes.js] использует настроенный [client.js] → летит запрос на бэк
      │
      ▼
[Компонент HomeDetail.vue] получает данные и вызывает [useBooking.js] → setSelectedHomeId(1)
      │
      ▼
[BookingForm.vue] (находится в App.vue) видит изменение в [useBooking.js] и подставляет ID в форму
```

### Ключевые концепции для базы знаний:
1. **Axios Instance & Interceptors** — централизация сетевых запросов и глобальная обработка ошибок.
2. **HTML5 History Mode** — создание SPA с нормальными URL без решёток (`#`) для SEO.
3. **Dynamic Routing & Catch-All** — безопасная обработка неизвестных URL.
4. **Composable as Shared State** — лёгкая альтернатива глобальным сторам (Pinia) для передачи данных между несвязанными компонентами.
