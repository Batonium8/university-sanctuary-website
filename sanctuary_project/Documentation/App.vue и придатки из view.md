# Архитектура приложения: App.vue и View-обёртки

Эти файлы образуют **скелет** приложения. Если компоненты (Header, Rooms, BookingForm) — это «органы», то эти файлы — «туловище», которое их соединяет.

---

## 1. App.vue — Корневой компонент (Layout)

Это **точка входа**. Vue всегда начинает рендеринг с этого файла.

### Назначение
`App.vue` задает **глобальный макет** (Layout), который остается неизменным, пока пользователь ходит по сайту.

```vue
<template>
  <div class="flex flex-col min-h-screen">
    <Header />
    <main class="grow">
      <router-view />
    </main>
    <Footer />
  </div>
</template>
```

### Разбор логики

1.  **`flex flex-col min-h-screen`**:
    *   Делает приложение флекс-контейнером на всю высоту экрана.
    *   **Зачем?** Чтобы футер всегда был внизу, даже если на странице мало контента (например, страница 404).

2.  **`<main class="grow">`**:
    *   Класс `grow` (flex-grow: 1) заставляет центральный блок занимать всё свободное пространство.
    *   Он «расталкивает» Header вверх, а Footer вниз.

3.  **`<router-view />`**:
    *   Это **слот для маршрутов**. Сюда Vue Router подставляет компонент, который соответствует текущему URL.
    *   **Теория:** Когда ты переходишь с `/` на `/rooms/1`, Vue **не перезагружает** `App.vue`. Он просто уничтожает старый компонент внутри `<router-view>` (HomeView) и монтирует новый (HomeDetailsView). Header и Footer при этом остаются висеть в памяти — это экономит ресурсы и сохраняет их состояние (например, открытое мобильное меню не закроется само по себе).

## 2. Паттерн «View-обёртка» (Container Component)

Файлы `HomeDetailsView.vue`, `ServiceView.vue`, `DoctorView.vue` выглядят очень простыми. Это не случайно.

### Архитектурная теория: Container vs Presentational
В современной разработке (особенно в React и Vue) принято разделять компоненты на два типа:

| Тип | Пример в проекте | Отвечает за |
| :--- | :--- | :--- |
| **Container (Умный)** | `HomeDetailsView.vue` | **Композицию и маршрутизацию.** Решает, *какие* блоки показать на странице. Знает про `<router-view>`, про глобальные формы. |
| **Presentational (Глупый)** | `HomeDetail.vue`, `BookingForm.vue` | **Отображение и логику данных.** Знает, как нарисовать карточку или отправить форму. Не знает, где он находится (на главной или в подвале). |

### Зачем так делать?
**Переиспользование.**
Посмотри на `BookingForm.vue`. Он используется сразу в трёх местах:
1.  `HomeView` (главная)
2.  `HomeDetailsView` (страница номера)
3.  `ServiceView` / `DoctorView` (страницы услуг/врачей)

Благодаря тому, что `BookingForm` «глупый» (не знает, где он лежит), мы можем просто импортировать его в любую View-обёртку, и он будет работать. Если бы логика формы была «зашита» в `HomeView`, нам пришлось бы дублировать код для других страниц.

## 3. Разбор конкретных View

### HomeDetailsView.vue (и аналогичные Service/Doctor)

```vue
<template>
  <HomeDetail />
  <div id="bookings">
    <BookingForm />
  </div>
</template>
```

**Ключевой момент:** Обёртка `<div id="bookings">`.
*   **Проблема:** Кнопка «Записаться» в Header должна скроллить к форме.
*   **Решение:** Мы даем блоку с формой уникальный ID. Теперь неважно, на какой мы странице — скрипт скролла из Header найдет этот ID и плавно прокрутит к нему.
*   **Композиция:** Здесь мы собираем «детальную информацию» + «форму захвата». Это классический паттерн Landing Page для конкретного продукта.

### HomeView.vue

```vue
<template>
  <HeroSection ... />
  <About />
  <Services />
  <Doctors />
  <Rooms />
  <div id="bookings">
    <BookingForm />
  </div>
  <FAQ />
</template>
```

**Роль:** Это просто **список секций**.
*   Здесь нет сложной логики, только импорт и расстановка компонентов в нужном порядке.
*   **ID `bookings`**: Здесь он тоже есть. Это гарантирует, что якорная ссылка `/#bookings` сработает и на главной, и на внутренних страницах (ведь структура `div id="bookings" > BookingForm` идентична).

## 4. Как работает переход между страницами (Сценарий)

Представим, пользователь кликнул по карточке врача.

1.  **Действие:** Клик по `<router-link :to="{ name: 'StaffDetail', params: { id: 5 } }">`.
2.  **Router:** Vue Router видит имя `StaffDetail`, смотрит в `router.js` и понимает: «Ага, мне нужно показать компонент `DoctorView.vue` и передать ему параметр `id=5`».
3.  **App.vue:** Видит изменение маршрута.
    *   `Header` — стоит на месте.
    *   `Footer` — стоит на месте.
    *   `<router-view>` — удаляет `HomeView` и монтирует `DoctorView`.
4.  **DoctorView (Обёртка):** Монтируется. Внутри себя она видит теги `<DoctorDetail />` и `<BookingForm />`.
5.  **DoctorDetail (Компонент):** Запускает `onMounted`, берет `id=5` из роута, идет в API за врачом.
6.  **BookingForm (Компонент):** Просто рисуется. Она ждет действий пользователя.

**Итог:** View-файлы — это **клей**. Они связывают маршруты (URL) с конкретным набором UI-компонентов, обеспечивая единообразие структуры (везде есть форма брони внизу) и переиспользование кода.
